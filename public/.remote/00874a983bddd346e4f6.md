ローカルLLMを「バッチ処理エンジン」として使う——Nemotronで自分のデータから成果物を自動生成する設計

はじめに

AIとの対話が日常になった今、ローカルLLMの使い道として最も多いのは「チャットUIで会話する」か「RAGで社内文書に質問する」のどちらかでしょう。もちろんそれ自体に価値はありますが、自分のPC上で動くGPUの計算能力をそれだけに使うのは、少しもったいない気がしていました。

半年ほど開発を続けるなかで、ローカルLLMにはもっと別の使い方があることに気づきました。対話相手としてではなく、処理パイプラインの一部品として組み込みます。一晩放置してバッチ的に走らせ、朝起きたら成果物ができている。そういう使い方です。

この記事では、NVIDIAのNemotronをvLLMで動かし、自分の開発環境に眠っているデータを読み取り専用でスキャンし、分析結果をテキストファイルとして出力する仕組みについて書きます。大規模なフレームワークもクラウドAPIも使いません。Pythonスクリプトがlocalhost:8000にHTTPリクエストを投げるだけの、極めてシンプルなアーキテクチャです。


Nemotronとは何か

NVIDIAが公開しているNemotronは、効率性と精度を両立させたオープンモデルファミリーです。重みもトレーニングデータもレシピもオープンで、商用利用も可能です。特にNano系のモデルはMamba2ベースのハイブリッドアーキテクチャを採用しており、パラメータ数に対して高い精度を実現しています。

今回使っているのは Nemotron-Nano-9B-v2-Japaneseです。2026年2月にNVIDIAが公開した日本語特化バリアントです。このモデルの出自は「9Bの小型モデル」ではありません。

まず、NVIDIAがNemotron-Hアーキテクチャ（Mamba-2とTransformerのハイブリッド）でスクラッチから約20兆トークンで学習した12Bパラメータのベースモデルがあります。これをGPU 1枚で動くサイズにプルーニングして9Bに圧縮したのがNemotron-Nano-9B-v2-Baseです。そこからDeepSeek R1やQwen3-235Bの推論トレースを合成データとしてポストトレーニングし、英語版のNemotron-Nano-9B-v2が完成します。モデルカードには「Improved using Qwen」と明記されており、NVIDIA単独の技術ではなくオープンソースコミュニティの成果を積極的に取り込んでいます。日本語版はさらにここからNemotron-Personas-Japanデータセットを使い、日本語の継続事前学習とSFTを重ねたものです。Nejumiリーダーボード4では10B以下モデルで1位を獲得しています。

9Bパラメータという軽量サイズながら、テキストの分類・構造化・要約といったタスクを実用的にこなせます。RTX 5090の上でvLLMを使って動かしていますが、推論速度は十分で、100件程度のバッチ処理なら1時間程度で終わります。

重要なのは、vLLMがOpenAI互換のAPIエンドポイントを提供してくれる点です。localhost:8000/v1/chat/completions にPOSTリクエストを投げれば、OpenAIのAPIとまったく同じJSON形式でレスポンスが返ってきます。つまり、スクリプト側から見れば「URLがlocalhostか api.openai.com かの違い」でしかありません。環境変数を一つ切り替えるだけで、ローカルとクラウドを自由に行き来でき、汚染を防止します。


基本設計：ファネルパイプライン

この仕組みの設計思想は単純で、以下の3段階で成り立っています。

Phase 1として、Pythonスクリプトでローカルのファイルシステムを読み取り専用でスキャンし、必要な情報だけを抽出します。数百ギガバイトのデータから数メガバイトのテキストに圧縮する工程です。

Phase 2として、抽出したテキストをNemotronのAPIに投げ、分類・構造化・テンプレート化などの「言語的な処理」を行います。出力は.txtや.jsonファイルです。

Phase 3として、生成された成果物をClaude Code（Opus 4.6）で検証・修正します。Nemotronの出力が実際のファイルシステムと一致しているかを、Claude Codeが直接確認して校正します。

このファネル構造にはいくつかの意味があります。

まず、LLMが実際に処理するデータ量を3桁から4桁削減できます。1テラバイトのデータを直接LLMに食わせるのは、ローカルでもクラウドでも非現実的です。Phase 1のPythonスクリプトが賢くフィルタリングすることで、LLMには本当に必要な情報だけが渡ります。

次に、Phase 2のNemotron処理はローカルで完結するため、電気代以外のコストがかかりません。仮にすべてをクラウドAPIで処理した場合との差は、控えめに見積もっても100倍程度になります。月に何度も回すバッチ処理であれば、年間で相当な差になるでしょう。

そして、Phase 3で高性能モデルによるファクトチェックを入れることで、ローカルLLMの弱点であるハルシネーションを補完しています。作業量の95%をローカルで処理し、残り5%の検証だけをClaude Codeに任せます。安い労働力で大量に処理し、高いコストは品質保証にだけ使います。


何を作ったか

この設計で実際に動いているスクリプトを紹介します。どれも同じアーキテクチャに従っています。

1つ目は、プロンプトカタログ生成です。projects配下の全.pyファイルをスキャンし、AIのAPIに送っているプロンプト文字列を正規表現で抽出します。「content」キーやsystem_prompt変数の代入文など、複数のパターンに対応しています。抽出したプロンプトをNemotronに送り、用途カテゴリ（コード生成、要約、分析、分類など）に分類させ、変数部分を{placeholder}に置き換えたテンプレート版を生成させます。出力は再利用可能なプロンプトのカタログです。自分のコードベースから「自分のプロンプトパターン集」が自動的にできあがります。

2つ目は、コードテンプレート生成です。Project配下の全コードファイルをスキャンし、今後の類似Projectに使えるコードセットを生成します。

3つ目は、環境ルールブック生成です。OS情報、Pythonバージョン、nvidia-smiの出力、インストール済みパッケージ一覧、プロジェクト構造、設定ファイルなどをコマンド実行で収集し、その事実データをNemotronに渡して開発環境のルールブックに整形させます。「uvを使うこと」「CUDAバージョンはこれ」「ポート割り当てはこう」といったルールが、実際の環境データに基づいて生成されます。各ルールには根拠の参照を付けさせているため、Phase 3でClaude Codeが照合しやすいです。

4つ目は、ローカルデータ横断分析です。README.md群からプロジェクトポートフォリオを生成する機能、Git設定ファイル（.gitconfig、.gitignore、SSH config、gh CLI設定）を横断収集してGit運用ルールブックを作る機能、そしてClaude Codeが実行したbashコマンドの履歴とシェルヒストリを集計してコマンド辞典を作る機能の3つを1スクリプトにまとめています。

5つ目は、デッドコード・不要ファイル検出です。空ファイル、ゴミファイル（.log、.bak、.tmp）、MD5ハッシュによる重複検出、孤立した__init__.pyの検出をファイルレベルで行い、さらにAST解析で未使用importや未呼び出し関数を検出します。クロスリファレンスで他ファイルから参照されているものは除外する仕組みも入れました。肥大化したディレクトリや放置された.venv、キャッシュ残骸も集計します。

6つ目は、Claude Codeのプラグイン・スキル分析です。~/.claude.jsonや.mcp.jsonからMCPサーバーの設定を収集してリスト化し、~/.claude/skills/配下のSKILL.mdを読み込んで既存スキルを棚卸しし、CLAUDE.mdの内容も収集します。そのうえでNemotronに「足りていないスキル」を提案させ、実際のSKILL.md形式で草案を出力させます。生成されたスキルファイルは~/.claude/skills/にそのままコピーできます。


APIという設計の意味

ここまで読んで気づいた方もいるかもしれませんが、これらのスクリプトが行っているのは「ファイルを読む」「HTTPリクエストを投げる」「テキストファイルを書く」の3つだけです。既存環境には一切手を加えません。pipで何かをインストールする必要もなければ、データベースのスキーマを変更することもありません。vLLMのプロセスを停止すればLLM側も完全にクリーンに戻ります。

これが可能なのは、ローカルLLMの推論サーバーが「API」という抽象化レイヤーを提供してくれるからです。vLLMにせよSGLangにせよllama.cppにせよ、どれもHTTPサーバーとしてエンドポイントを立て、OpenAI互換のJSON形式でやり取りします。スクリプト側はrequestsライブラリでPOSTリクエストを投げるだけです。

この「疎結合」が生み出す自由度は大きいです。モデルを差し替えたければ環境変数のMODEL名を変えるだけです。推論サーバーをvLLMからSGLangに変えてもスクリプトの修正はゼロです。クラウドAPIに切り替えたければURLを変えるだけです。処理の本体であるPythonスクリプトは、LLMの実装詳細を一切知らなくて済みます。

同じOpenAI互換APIはOllamaでも使えます。実際、スクリプトのコードは一行も変えずにOllamaでも動きます。ではなぜvLLMを選んだのか。正直に言えば、並列処理の恩恵はほとんどありませんでした。RTX 5090とはいえ、9Bモデルを載せればVRAMの余裕は限られ、同時リクエスト数は1に絞り、その代わりmax_tokensを最大に確保することで出力の完走率を上げました。それでもvLLMを使い続けている理由は、推論パラメータの細かい調整と、サーバープロセスとしての安定性です。一晩放置するバッチ処理では、途中でプロセスが落ちないことが何より重要になります。


なぜNemotron＋vLLMなのか——NVIDIAスタックの実態

「Nemotron、CUDA、TensorRT、vLLM——全部NVIDIAエコシステムだから相性抜群」と思いがちですが、実態はもう少し複雑です。

理想的なスタックのイメージはこうでしょう。

  Nemotron（モデル）
    ↓ NVIDIA製、NVIDIA GPU向けに最適化
  TensorRT-LLM（推論最適化）
    ↓ 量子化・カーネル融合・KVキャッシュ最適化
  vLLM（サービング）
    ↓ PagedAttention、スケジューリング
  CUDA（GPU計算基盤）
    ↓
  RTX 5090（ハードウェア）

しかし実際に動いている構成はこうです。

  Nemotron → vLLM → PyTorch → CUDA → RTX 5090
             （TensorRTは経由していない）

vLLMのデフォルトバックエンドはPyTorch＋独自CUDAカーネル（FlashAttention等）であり、TensorRTは使っていません。NVIDIAはTensorRT-LLM＋Triton Inference Serverという独自のサービングスタックを持っていますが、これはvLLMとは競合関係にあります。TensorRT-LLMを使えば推論速度は上がる可能性があるものの、セットアップの複雑さが跳ね上がり、モデル変換（エンジンビルド）が必要になるため、モデルを気軽に差し替えられなくなります。

さらに興味深いのは、Nemotron NanoがMamba2＋MoEのハイブリッドアーキテクチャを採用していることです。標準的なTransformerとは異なるため、vLLMの対応状況に依存します。vLLMは近年Mamba系のサポートを追加していますが、最適化の深さはTransformerほどではありません。TensorRT-LLMでも同様に、ハイブリッドアーキテクチャのサポートは発展途上にあります。つまり「NVIDIA統一スタック」というイメージとは裏腹に、実際に効いているのはCUDAレベルの最適化とvLLMのメモリ管理であって、TensorRTの恩恵は受けていません。それでも十分実用的な速度が出ているのは、9Bという小さいモデルサイズとRTX 5090の32GB VRAMのおかげです。


Thinking出力への対処

一つ、実装上のはまりどころがありました。Nemotron Nanoはデフォルトでreasoning（thinking）モードが有効になっています。APIのレスポンスに「Okay, let's tackle this...」という思考過程がそのまま含まれ、その後にJSONが続きます。プロンプトカタログの生成で106件中87件がパース失敗したのは、これが原因でした。

対処は力技になりますが確実です。レスポンスのテキストから思考過程を除去する関数を共通化し、全スクリプトで使い回します。<think>タグがある場合はその中身を除去し、タグがない場合はテキスト中の最後の{...}ブロックを探してJSONとしてパースを試みます。リクエスト時にenable_thinking: falseを指定する方法もありますが、サーバー実装によっては無視されるため、パーサー側で対処するのが最も堅牢です。


コスト

実際の数字で比較します。

プロンプトカタログの106件分類をすべてクラウドAPIで行った場合、1件あたり数千トークンの入出力で、全体では十数ドル程度かかります。これに環境ルールブック、README分析、コマンド辞典、デッドコード検出、スキル分析を加えれば、1回の実行で数十ドルになります。

ローカルNemotronの場合、コストは電気代だけです。RTX 5090で9Bモデルを走らせる電力消費はごく僅かで、1回のバッチ実行でも数円程度です。Claude Codeによる最終チェックだけがAPI費用として発生しますが、数千行のテキストを読んで修正するだけなので数百円で済みます。

月に数回、これらのスクリプトを回すとすれば、年間のコスト差は数十万円規模になりえます。ローカルGPUの初期投資はあるものの、開発やゲーム、AI推論など他の用途と共有できることを考えれば、バッチ処理専用のコストとしては事実上ゼロに近いです。


データがローカルにあることの意味

もう一つ、見逃せない点があります。開発環境のデータには、外部に送りたくないものが含まれていることが多いです。API キー、内部プロジェクトのコード、クライアントとの契約に関わるファイル、個人的な投資情報。これらをクラウドAPIに送信するのは、技術的には可能でも、実務的には躊躇することがあります。

ローカル完結のパイプラインでは、データは自分のマシンから一切出ません。ファイルシステムのスキャンも、LLMへの推論リクエストも、すべてlocalhost上で完結します。Phase 3でClaude Codeを使う際も、送信するのはNemotronが生成した「成果物」であって、元のソースデータではありません。


設計パターンとしての汎用性

振り返ってみると、ここで使っているパターンは極めて単純です。

「ファイル読み取り → 抽出・集約 → localhost:8000 → sqlite」

データソースを差し替えるだけで、いくらでもバリエーションを作れます。実際、この記事で紹介した5つのスクリプトは、どれもこの型に従っています。新しいスクリプトを作るとき、考えるべきは「何のデータを」「どう抽出して」「LLMに何を聞くか」の3つだけです。LLMの接続周りやファイル出力のコードはほぼコピペで済みます。

run_all.pyというラッパースクリプトも作りました。複数のスクリプトを順次実行し、進捗をで管理します。中断しても次回は未完了のスクリプトから再開します。すべて完了すると、Claude Codeに渡すためのサマリーが表示されます。一晩放置する運用を前提にした設計です。


今後の拡張

このパイプラインは、データソースさえあれば横に広げ続けられます。いくつかアイデアがあります。

Chromeの閲覧履歴はSQLiteファイルとしてローカルに保存されています。これを読み込んで検索クエリの傾向を分析すれば、「いつ何を調べて、どのコミットに繋がったか」という開発活動のタイムラインが作れるかもしれません。

vLLMのアクセスログ自体も分析対象になります。どのモデルをいつどれくらい使ったか、推論時間の傾向はどうか。GPU利用効率の最適化に繋がるデータが、ログの中に眠っています。

判例データベースの横断分析も考えています。3万件の判例データがSQLiteに入っているので、特定のテーマに関する判例を横断的に抽出し、論点別のサマリーを自動生成します。Nemotronの日本語能力がどこまで法律文書に通用するかの検証にもなります。

US特許データベースも400万件近くまで成長してきているので、これは英語ですが同様に整理できる可能性があります。なお、どんなに安いAPIでも1処理で1000ドル以上かかるためメリットが大きいです。

もう少し先の話として、このパイプラインの各段階をMCPサーバー化すれば、Claude Code自体が「このデータをNemotronで分析して」と指示するだけで全工程が自動実行される世界も見えてきますし、定期的に巡回することもできます。ローカルLLMとクラウドLLMが、それぞれの得意分野で役割分担しながら協調します。そういう構成は、まだあまり実例を見ません。
